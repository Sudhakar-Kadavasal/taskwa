# TaskWA — working context for AI sessions

Read this first. It captures the project state, conventions, and decision
history from the sessions that built v1.0 → v1.6.3 (July 2026), so any new
session can continue without re-deriving context.

## What this is

TaskWA: self-hosted WhatsApp task manager for a small team. Daily numbered
digests over WhatsApp; members reply "1 done / 1 block <reason>"; admin web
dashboard; runs on the owner's (Sudhakar Kadavasal, "Sk") **personal
WhatsApp number** via WAHA. Python 3.12 / FastAPI / SQLAlchemy / SQLite
(WAL) / APScheduler / Jinja2 / Docker Compose. GitHub:
`sudhakar-kadavasal/taskwa` (public). Brand: **TaskWA** (capital WA),
**Mint Ledger** palette everywhere (docs, dashboard, flyer).

## Locations & workflow (important)

- **Single source of truth: `~/github/TaskWa`** (this repo). All builds and
  edits happen here, on `main`. Never create parallel copies.
- **The runtime is a SEPARATE folder** containing `.env` and `data/`
  (find it: `docker inspect taskwa-app-1 --format
  '{{ index .Config.Labels "com.docker.compose.project.working_dir" }}'`).
- Deploy ritual after every change: commit+push here, then in the runtime
  folder `git pull && docker compose up -d --build app`. **"Committed" ≠
  "deployed"** — several "bugs" were just stale runtimes.
- **Sandbox git quirk** (Cowork sessions): the sandbox cannot delete files
  on the mount, so `.git/index.lock` / `tmp_obj_*` accumulate and block
  commits. The user runs locally:
  `find .git \( -name "*.lock" -o -name "tmp_obj_*" \) -delete`
  then commits. Stage everything with `git add -A`; the user commits/pushes.
- Tests: `python -m pytest tests/ -q` — 60 passing as of v1.6.3. Always run
  before handing over; add regression tests for every live bug.
- iCloud is for backups only, never live data (SQLite + sync = corruption).
- Docs PDFs are generated by `tools/docgen/` (see its README).
  `docs/TaskWA-Flyer.pdf` is the user's approved file — never regenerate.

## Architecture map

- `app/webhook.py` — WAHA webhook (`?token=WEBHOOK_SECRET`). Accepts events
  `message` and `message.any` (**message.any is required** so the owner's
  own `fromMe` messages arrive). LID resolution, personal-mode filtering,
  idempotency (ProcessedMessage), catch-all: **never returns 500 to WAHA**
  (a 500 causes a retry storm — one bug = message every 2 s).
- `app/commands.py` — command parser/dispatcher. Verbs: done, reopen,
  in progress/wip, block(er/ed), unblock, cancel. `/add` (with @mentions,
  !high/!low, date words, Y/N PendingConfirm), `/mytasks`, `/list`, `/help`.
  Reply object carries text/react/alert_admin/extra_sends/unmatched.
- `app/engine.py` — create_task, change_status (permission matrix, see
  below), digest ref resolution (positional numbers → serials).
- `app/digest.py` — digest builders + "Waiting on you" section +
  alert_admins (with exclude-dedupe).
- `app/broadcasts.py` — Broadcasts: owner's verbatim scheduled messages
  (no headers/numbers), per-broadcast **pinned timezone**, {date}/{day}
  placeholders, 20–45 s pacing.
- `app/scheduler.py` — digest crons (6 h catch-up on cold boot), broadcast
  crons (**missed broadcasts deliberately skipped**), health check /5 min,
  nightly backup 02:30 (keep 14), purge 03:00, acceptance-expiry sweep
  /5 min (30-min auto-accept).
- `app/waha.py` — WAHA client. `send_text` is THE outbound choke point:
  canonicalises `@lid` → `phone@c.us`, enforces the **hard allowlist**
  (only active registered members/groups ever receive anything), dry-run,
  hourly cap. `paced_send` (3–8 s digests) under a global batch lock.
- `app/models.py` — Member, Group, Task (global never-reused serials,
  waiting_on_id), StatusEvent (audit), Setting, ProcessedMessage,
  PendingConfirm (UNIQUE member_id — delete stale rows before insert!),
  MessageLog (sent/failed/blocked/dryrun), DigestRef, GroupDigestRef,
  Broadcast (tz pinned at save).
- `app/main.py` — startup: create_all + `_migrate()` (additive SQLite
  ALTERs — add new columns here) + `_stamp_broadcast_tz()`.
- `app/ui.py` — dashboard (admin login only). Mint Ledger CSS in
  `app/static/style.css`.
- `scripts/autostart-*.{sh,ps1}` — unattended restart chain
  (docs/UNATTENDED.md).

## Invariants — do not break these

1. **Silence to strangers.** Unregistered senders/groups get nothing, ever.
   The allowlist in `send_text` is the final guarantee; nothing bypasses it.
2. **Personal-number mode**: bot never replies to non-commands; owner's
   private chats never touched; owner's digest goes to "Message Yourself".
   Unmatched replies are suppressed in groups and personal mode.
3. **No double messaging.** One event → one message per person. Group-posted
   tasks are announced in the group digest, never also DM'd. Admin alerts
   exclude anyone already getting a specific notice for the same event.
4. **Ban-risk hygiene**: hourly cap, randomized pacing, global send lock,
   broadcasts 20–45 s apart. This runs on the owner's personal number.
5. **Work never dies silently**: decline returns the task to its creator
   (admin fallback); cancel is creator/admin-only; every change is a
   StatusEvent in the audit trail.
6. Digest numbers are positional per-member/per-group maps; **global task
   serials always work as fallback** in any reply.

## WAHA operational knowledge (hard-won)

- Apple Silicon needs `WAHA_TAG=arm` in .env (`latest` is amd64-only).
- Empty `WAHA_API_KEY` → WAHA generates its own → all 401s. Set a fixed key.
- Webhook events must include `message.any` (fromMe) — changing it needs
  `docker compose up -d --force-recreate waha`.
- Privacy-enabled accounts appear as `...@lid` in groups AND DMs. Inbound:
  resolve via `/api/{session}/lids/{lid}`; outbound: `send_text`
  canonicalises. LIDs resolve only if the number is visible (contact, or
  bot number is group admin). First message after restart may need the
  group-list warmup; second message works.
- Session states: WORKING / SCAN_QR_CODE / STARTING / FAILED / UNREACHABLE.
  Pairing survives restarts (data/waha/). `list_groups` needs a 120 s
  timeout. Health page banner reads a cache refreshed on every live check.
- Credentials pasted into chat are burned — generate fresh ones.
